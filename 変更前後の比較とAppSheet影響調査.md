# 変更前の動作と変更後の比較 + AppSheet影響調査

---

## 第1部：変更前はどう動いていたか

### 全体の流れ（変更前後で同じ）

```
AppSheetで単価登録（メーカー・商社・品名・単価変動を入力）
  ↓
「単価一括登録」シートに1行追加される
  ↓
GASを手動実行
  ↓
「単価履歴」シートに展開して書き込み
```

### 変更前のGASの具体的な動き

#### ステップ1: 未転記データの取得
- 「単価一括登録」シートの全データを取得
- N列（転記済）が `1` でない行だけ抽出

#### ステップ2: 重複チェック用のデータ準備
- 「単価履歴」シートの**全データを1回読み込み** → ここまでは同じ

#### ステップ3: 商社×品名の展開ループ
未転記の行ごとに以下を繰り返す：
1. 商社をカンマで分割（例：「阪和, 豊通」→「阪和」「豊通」）
2. 品名をカンマで分割（例：「HS1, HS2」→「HS1」「HS2」）
3. 全組み合わせ（2×2＝4通り）をループ
4. **組み合わせ1つごとに「単価履歴」の全データを読み直して前回単価を検索** ← ここが遅かった

#### ステップ4: 書き込み
- 履歴シートに一括書き込み（ここは同じ）
- オートフィルで**全行**に数式コピー ← 5,000行以上にコピーで遅かった
- 転記済フラグを**1行ずつ**書き込み ← 30行なら30回通信で遅かった

#### ステップ5: 行番号の特定
- A列のIDを**別途全件取得**して、`indexOf`で行番号を検索 ← 余計な通信＋遅い検索方法

### 変更前に潜んでいたバグ

#### バグ1: 重複チェックが効いていなかった
重複チェックで使うキーの形式が、入力側と履歴側で違っていた。

```
入力側のキー: 「2024/11/14」＋「中山製鋼:阪和 , 豊通」＋「HS2」＋「阪和」
                                  ↑カンマ区切り全体

履歴側のキー: 「Thu Nov 14 2024...」＋「中山製鋼:阪和」＋「HS2」＋「阪和」
               ↑日付の形式も違う      ↑個別の商社
```
→ 永遠にマッチしない → 重複チェックが実質無効 → 転記済フラグだけで重複を防いでいた

#### バグ2: 前回単価が見つからなかった（複数商社選択時）
前回単価を探すキー（処理列2）も同じ問題。
「中山製鋼:阪和 , 豊通HS2」で検索するが、履歴には「中山製鋼:阪和HS2」で保存されているのでヒットしない。
→ 前回単価が常に0になる → ただし数式（オートフィル）が上書きするので最終的な表示には影響しなかった

## 第2部：変更前と変更後の比較表

| 処理 | 変更前 | 変更後 |
|------|--------|--------|
| 履歴データ取得 | 組み合わせ数だけ繰り返し読込（数千回） | 最初に1回だけ読込 |
| 前回単価の検索 | 毎回シートから全件読込 | メモリ上のMapで即座に検索 |
| 重複チェックのキー | 入力行のカンマ区切り全体を使用（バグ） | 展開後の個別商社で生成（修正済み） |
| 前回単価のキー | 同上（バグ） | 同上（修正済み） |
| 日付のフォーマット | 履歴側はDate型をそのまま結合（バグ） | 両方ともyyyy/MM/dd形式に統一 |
| 転記済フラグ書込 | 1行ずつ setValue | N列を一括で読み書き |
| オートフィル範囲 | 2行目〜最終行（全行） | 新規追加行のみ |
| オートフィルのシート取得 | openByIdで再取得 | 引数で再利用 |
| スプレッドシートID | コード内に複数箇所 | 先頭のグローバル定数1箇所 |
| 行番号の特定 | A列を別途取得＋indexOf検索 | フィルタ時に行番号を保持 |
| タイムゾーン取得 | ループ内で毎回取得 | ループ外で1回キャッシュ |

## 第3部：AppSheetへの影響調査

### 結論：影響なし

### 調査内容

#### 1. AppSheetが読み書きするシートとGASの関係

```
AppSheet → 「単価一括登録」に行を追加（GASは読むだけ・転記済フラグのみ書込）
GAS → 「単価履歴」に行を追加
AppSheet ← 「単価履歴」を表示（登録履歴ビュー、単価推移グラフ等）
```

GASが書き込むのは「単価履歴」シートだけ。AppSheetが入力する「単価一括登録」の内容は変えない（転記済フラグを除く）。

#### 2. 「単価履歴」に書き込むデータ形式の変化

| 列 | 変更前 | 変更後 | AppSheet影響 |
|----|--------|--------|-------------|
| A: id | UUID | UUID（同じ） | なし |
| B: 日付 | yyyy/MM/dd | yyyy/MM/dd（同じ） | なし |
| C: メーカー | そのまま | そのまま（同じ） | なし |
| D: 商社 | 個別商社名 | 個別商社名（同じ） | なし |
| E: 処理列1 | スクリプトが値を書込→数式で上書き | 同じ | なし |
| F: 品名 | そのまま | そのまま（同じ） | なし |
| G: 処理列2 | スクリプトが値を書込→数式で上書き | 同じ | なし |
| H: 単価 | スクリプトが値を書込→数式で上書き | 同じ | なし |
| I: 単価変動 | そのまま | そのまま（同じ） | なし |
| J: 前回単価 | スクリプトが値を書込→数式で上書き | 同じ | なし |
| K: スポット | そのまま | そのまま（同じ） | なし |
| L: スポット期間 | そのまま | そのまま（同じ） | なし |
| M: 備考 | 空欄 | 空欄（同じ） | なし |
| N: 最新フラグ | 空欄→数式で上書き | 空欄→数式で上書き（同じ） | なし |
| O: タイムスタンプ | そのまま | そのまま（同じ） | なし |
| P: 検索列 | スクリプトが値を書込→数式で上書き | 同じ | なし |

**E, G, H, J, N, P列はオートフィルの数式が最終値を決める**ため、スクリプトが書き込む中間値が変わっても最終結果は同じ。

#### 3. AppSheetのビュー・アクションへの影響

| AppSheet機能 | 参照先 | 影響 |
|-------------|--------|------|
| 登録履歴ビュー | 単価履歴の日付・メーカー・商社・品名・単価 | なし（同じデータ） |
| 単価推移グラフ | 単価履歴の日付・単価 | なし（数式が計算） |
| 単価検索 | 単価履歴の検索列(P) | なし（数式が計算） |
| 単価一括登録_Form | 単価一括登録シート | なし（GASは読むだけ） |
| 一括単価登録アクション | 前回単価列にSET_COLUMN_VALUE | なし（GASとは独立） |
| 最新単価表 | 単価履歴を参照 | なし（同じデータ） |

#### 4. 唯一のプラスの変化

バグ修正により**重複チェックが正しく機能する**ようになったため、同じメーカー×商社×品名×日付の組み合わせが二重に履歴に入ることがなくなった。
→ AppSheet上の表示データがより正確になる（プラスの影響）。
